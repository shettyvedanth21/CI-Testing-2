## 1. What this system is

Shivex-Main (product name **FactoryOPS**, customer-facing brand **Cittagent**)
is a multi-tenant SaaS for industrial monitoring — primarily compressors and
energy meters — built on Docker Compose, Python (FastAPI + SQLAlchemy +
Alembic), MySQL, Redis, InfluxDB, EMQX (MQTT), MinIO/S3, Next.js 16, and
Expo (React Native). A single deployment runs every service in containers on
one network (`energy-net`); all first-party app services publish through
GHCR and are pulled by tag, with a documented source-build fallback for
emergencies.

The product surface:

- **Operator dashboard** (`ui-web`, Next.js 16, port 3000)
- **Operator mobile** (`shivex-mobile`, Expo Router)
- **Device telemetry ingest** (MQTT → data-service → Redis streams → InfluxDB)
- **Live fleet stream** (SSE/WebSocket from device-service)
- **Anomaly + degradation scoring** (machine-health pipeline in device-service)
- **Rule engine + alerts + notifications** (email/SMS/WhatsApp)
- **Async analytics jobs** (forecasting, failure prediction, ensemble, retrain)
- **Async reporting** (consumption, cost, demand, comparison, overtime,
  reactive, hidden overconsumption, load factor, power quality) → PDF + CSV
- **Waste / idle analysis** with PDF output
- **Data export** to S3/MinIO for downstream dataset consumers
- **Copilot** (LLM-backed) tenant-scoped chat over a curated read-only model
- **Per-org feature entitlements** (premium feature grants + role-feature
  matrix)
- **Multi-tenant everywhere** (every row carries `tenant_id`; SH-tenant
  hard-cuts ran in 2026-04 / 2026-05)

Networks, volumes, and image policy:

- One bridge network: `energy-net`.
- Persistent volumes: `influxdb_data`, `emqx_data`, `emqx_log`,
  `mysql_data` (local), `minio_data` (local), `prometheus_data`,
  `alertmanager_data`, `grafana_data`.
- **Local dev** uses `docker-compose.yml + docker-compose.local.yml` with
  source builds and adds MySQL + MinIO + a `local-device-simulator`.
- **Production** uses the base file only; images come from GHCR. EMQX is
  configured by `ops/emqx/production.base.hocon` mounted read-only. The
  `emqx-auth-bootstrap` one-shot creates the `internal:data-service` MQTT
  user with the value of `MQTT_PASSWORD` (idempotent: PUT if exists).
- **EMQX operational rule** (from `memory.md`): `ops/emqx/*.hocon` is the
  source of truth. Never edit runtime config from the EMQX dashboard for
  local validation. If MQTT behavior drifts from git config, remove the
  `emqx_data` volume and recreate the container so it boots from
  `local.base.hocon`.

## 4. Per-service deep dive

### 4.1 `auth-service` (port 8090)

- **Layout**: `app/{api,models,repositories,schemas,services}/`.
- **Routers** (`app/api/v1/`):
  - `auth.py` — login, refresh, logout, forgot/reset password,
    accept-invite, action-token status, `/me` (with entitlements).
    Refresh tokens are stored as `httponly` cookies (samesite from
    `REFRESH_COOKIE_SAMESITE`, secure from `REFRESH_COOKIE_SECURE`).
  - `orgs.py` — tenant CRUD, plant CRUD, user CRUD, feature-entitlement
    GET/PATCH, lifecycle endpoints.
  - `admin.py` — admin-only operations.
  - `platform_maintenance.py` — platform maintenance announcements CRUD
    and delivery metadata.
- **Bootstrap lifespan** (`app/services/bootstrap_service.py`):
  - `ensure_bootstrap_super_admin` — creates super admin from
    `BOOTSTRAP_SUPER_ADMIN_*` env on first boot.
  - `ensure_local_bootstrap_state` — when `LOCAL_BOOTSTRAP_ENABLED=true`,
    seeds tenant `SH00000001` / plant `demo-plant` / super admin.
  - `ensure_tenant_allocator_state` — initializes the tenant-ID sequence.
  - `token_cleanup_service.refresh_token_cleanup_svc` — periodic cleanup.
  - `platform_maintenance_delivery.platform_maintenance_delivery_worker` —
    background delivery loop.
- **Models** (`app/models/auth.py`):
  - `Organization` (id `String(10)` shape `SHxxxxxxxx`, slug unique,
    `is_active`, `entitlements_version`, `premium_feature_grants_json`,
    `role_feature_matrix_json`).
  - `Plant` (uuid, `tenant_id` FK cascade, name, location, timezone,
    `is_active`).
  - `User` (uuid, `tenant_id` nullable for `SUPER_ADMIN`, email indexed,
    password hash, role, lifecycle timestamps).
  - `UserPlantAccess` (many-to-many).
  - `RefreshToken`, `AuthActionToken` (with `AuthActionType` enum:
    `INVITE_SET_PASSWORD`, `PASSWORD_RESET`), `TenantSecurityAuditLog`.
  - `PlatformMaintenance*` (severity, status, email delivery status enums).
- **Rate limiting** is wired in `app/rate_limit.py` (slowapi-style with
  Redis backend) and applied per-route.
- **Hardening**: `SensitiveDataFilter` redacts `authorization`, `password`,
  `refresh_token`, etc. from logs. Errors pass through
  `_sanitize_validation_errors` to drop `input` payloads. CORS uses
  `cors.build_allowed_origin_hosts(settings.FRONTEND_BASE_URL)`.

### 4.3 `data-service` (port 8081)

- **Layout**: `src/{api,config,handlers,models,queue,repositories,services,utils,workers}/`.
- **Bootstraps in lifespan** (`src/main.py`):
  - `MQTTHandler`, `TelemetryService`,
  - `TelemetryPipelineWorker`, `OutboxRelayService`,
  - `DLQRetryService`, `ReconciliationService`,
  - `RetentionCleanupService`, `ensure_bucket_retention`.
  - `WebSocketTicketService.close_websocket_ticket_service` on shutdown.
  - Startup contract validated via `services.shared.startup_contract`.
- **MQTT handler** (`src/handlers/mqtt_handler.py`):
  - Subscribes to `settings.mqtt_topic`, `devices/+/telemetry` (legacy),
    and `+/devices/+/telemetry` (tenant-aware) at QoS 1.
  - Extracts `(tenant_id, device_id)` from the topic; if both topic and
    payload carry `tenant_id`, they must match or the message is dropped.
  - Resilient reconnect + loop-safe async handoff
    (`_loop`, `_client_lock`).
  - Validates `device_id` against `^[A-Za-z0-9_-]+$`.
- **Validation** (`src/utils/validation.py`):
  - `TelemetryValidator.validate_payload` — checks `device_id`, `timestamp`,
    and numeric fields. Non-numeric allowed: `tenant_id`, `schema_version`,
    `enrichment_status`, `device_metadata`, `enriched_at`. Returns
    `(is_valid, error_type, error_message)`.
- **Pipeline** (`src/workers/telemetry_pipeline.py`): durable Redis-stream
  stage workers (8 persistence, 4 projection, 4 broadcast, 6 energy, 8
  rules workers in `data-telemetry-worker`; another full set in
  `data-telemetry-worker-2`).
- **API** (`src/api/`):
  - `routes.py` — health/ready, MQTT stats, telemetry history, range
    queries, ingest (HTTP), batch ingest, internal active-tenant-ids.
  - `telemetry.py` — `GET /telemetry/{device_id}` and
    `GET /stats/{device_id}`.
  - `websocket.py` — `WS /ws/telemetry/{device_id}` and `GET /ws/stats`.
- **Models**:
  - `Telemetry` (InfluxDB shape), `TelemetryOutbox` (MySQL), `DLQMessage`
    (MySQL, see `init-scripts/mysql/*.sql`), `ReconciliationLog`.
- **Key env**: `TELEMETRY_INGEST_STREAM_NAME=telemetry_ingest_stream` and
  analogues for `projection`, `broadcast`, `energy`, `rules`,
  `dead_letter`. All have `MAXLEN=250000` and `BACKLOG_THRESHOLD=200000`
  (rules 10000, energy 5000). `TELEMETRY_HEALTH_LAG_*` warn/overload
  thresholds and `TELEMETRY_PROJECTION_OVERLOAD_*` are tunable.

## 5. Data plane details

### 5.1 MQTT topic model

```
Legacy:        devices/<device_id>/telemetry
Tenant-aware:  <tenant_id>/devices/<device_id>/telemetry
```

The data-service subscribes to both for backward compatibility. Tenant
mismatch between topic and payload → message is dropped (warning logged).

### 5.2 Per-device MQTT auth & ACL

- EMQX auth/ACL is **MySQL-backed** (`ops/emqx/local.base.hocon`,
  `ops/emqx/production.base.hocon`) against tables
  `device_mqtt_credentials` and `device_mqtt_acl` (managed by
  `device-service` Alembic migrations including
  `20260423_0001_device_mqtt_credentials`).
- `device-service` issues **hashed** passwords on demand and writes per-
  device ACL rows. Username format: `device:<tenant_id>:<device_id>`.
- The simulator / firmware flow:
  `credential_bootstrap.py` → `internal_service_auth.py` → connect →
  `mqtt_client.py` (QoS 1) → publish to the tenant-prefixed topic.
- An `emqx-auth-bootstrap` one-shot container creates/updates the
  `internal:data-service` MQTT user using the value of `MQTT_PASSWORD`.

### 5.3 Telemetry payload contract

Required: `device_id`, `timestamp` (UTC ISO-8601).
Numeric-only fields: anything not in the explicit non-numeric set
(`tenant_id`, `schema_version`, `enrichment_status`, `device_metadata`,
`enriched_at`) must be a number.

Canonical names:
- `voltage` (V), `current` (A), `power` (W), `power_factor` (0.0–1.0),
  `energy_kwh` (cumulative).
- Aliases accepted by the backend:
  - `current_l1..l3`, `phase_current`, `i_l1` → `current`
  - `voltage_l1..l3`, `v_l1` → `voltage`
  - `pf` → `power_factor`
  - `active_power`, `kw` → `power`
- `schema_version` should be `v1`.

Example:

```json
{
  "device_id": "COMPRESSOR-001",
  "timestamp": "2026-03-04T12:00:00Z",
  "schema_version": "v1",
  "voltage": 231.4,
  "current": 0.86,
  "power": 198.7,
  "power_factor": 0.98,
  "energy_kwh": 1245.337
}
```

### 5.4 Telemetry pipeline (data-service)

1. **MQTT bridge** (`MQTTHandler`) — paho client, multi-topic subscribe,
   tenant-id resolution, message handoff to async loop.
2. **Validation** (`TelemetryValidator`) — drops invalid messages, sends
   them to the dead-letter stream with the typed error
   (`missing_required_fields`, `invalid_timestamp`,
   `invalid_numeric_fields`, `validation_error`).
3. **Redis stage streams** (all with `MAXLEN=250000`):
   - `telemetry_ingest_stream` (8 persistence workers)
   - `telemetry_projection_stream` (4 projection workers)
   - `telemetry_broadcast_stream` (4 broadcast workers for SSE/WS)
   - `telemetry_energy_stream` (6 energy workers)
   - `telemetry_rules_stream` (8 rules workers)
   - `telemetry_dead_letter_stream` (DLQ)
4. **Outbox + DLQ** — `telemetry_outbox` MySQL table (enums
   `telemetry_outbox_target` ∈ {`device-service`, `energy-service`} and
   `telemetry_outbox_status` ∈ {`pending`,`delivered`,`failed`,`dead`})
   with retry/backoff and a circuit breaker
   (`OUTBOX_CIRCUIT_BREAKER_HALF_OPEN_MAX_CALLS=3`).
5. **Reconciliation** — `reconciliation_log` records `influx_ts` vs
   `mysql_ts` drift and the action taken; runs every
   `RECONCILIATION_INTERVAL_SEC=300`.

### 5.5 Backpressure knobs (memorize these)

- Telemetry health lag warn/overload: `30s / 90s`.
- Projection overload threshold: `20000` (max defers `40`, base
  `0.25s`–`8s`).
- Outbox pending warn/overload: `2000 / 10000`.
- Analytics queue reject threshold: `500`; per-tenant queued/active
  `25 / 8`.
- Reporting queue reject threshold: `5000`; per-tenant pending/active
  `25 / 4`.
- Rule-engine: per-tenant concurrency/queues similarly scoped.

### 5.6 Multi-tenant and feature entitlements

- Every tenant-scoped row carries `tenant_id`.
- `Organization.entitlements_version` is an integer; bumping it
  invalidates client caches of entitlement state.
- `premium_feature_grants_json` (list of premium features) and
  `role_feature_matrix_json` (dict role→features) override baseline per
  role. Baseline per role is in
  `services.shared.feature_entitlements.BASELINE_FEATURES_BY_ROLE`.
- `build_feature_entitlement_state(role, premium_feature_grants,
  role_feature_matrix, entitlements_version)` computes
  `effective_features_by_role`, `available_features`, etc.
- `MeResponse` returns this state so the UI can render
  `FeatureGate` / `LockedPremiumCard`.
- Hard-cuts to SH tenants ran across
  `device-service` (`20260411_0001_sh_tenant_id_hard_cut`),
  `auth-service` (`0008_hard_cut_sh_tenant_ids`),
  `rule-engine-service` (`20260411_0008_sh_tenant_id_hard_cut`),
  `waste-analysis-service` (`006_sh_tenant_id_hard_cut`).

  `ghcr.io/${GHCR_OWNER_LOWER:-cittagent}/${GHCR_REPO_LOWER:-shivex-main-1}-<service>:<tag>`

  `build:` for first-party services only. It does **not** add local
  MySQL/MinIO/simulator.

## 14. Operational gotchas (read these before changing anything)

1. **EMQX source of truth**: `ops/emqx/*.hocon`. A stale runtime
   override in `/opt/emqx/data/configs/cluster.hocon` once replaced
   the intended MySQL auth config and broke MQTT logins even though
   git was correct. Reset the `emqx_data` volume and recreate EMQX
   to recover.
2. **Analytics NOGROUP self-heal**: if the analytics Redis consumer
   group disappears, the worker detects `NOGROUP`, resets group
   readiness, recreates the group once, and retries — see
   `services/analytics-service/src/workers/job_queue.py`. Worker
   restarts alone are not enough.
3. **Tenant hard-cuts**: four services ran SH-tenant hard-cuts
   (`auth-service`, `device-service`, `rule-engine-service`,
   `waste-analysis-service`). Tenant data that still has a NULL
   `tenant_id` after a hard-cut will be rejected by repository
   filters; always run a backfill before deploying a hard-cut
   migration.
4. **Simulator cleanup**: `docker compose down -v --remove-orphans`
   does **not** remove standalone simulator containers. Use
   `simulatorctl.sh purge` or `preprod_validation.py --mode
   full-reset`.
5. **Redis `noeviction`** is intentional: durable queues must not
   drop jobs. Tune `REDIS_MAXMEMORY` if needed but do not switch to
   an evicting policy.
6. **Local UI build gate**: `npm run build` runs `lint:hooks`
   first; do not skip the hook-order lint.
7. **Migrations on startup**: every service runs Alembic on boot in
   its own service DB scope. Do not run schema migrations by hand
   against `ai_factoryops` while services are running.
8. **Feature gate contract**: every protected UI surface checks
   entitlements from `auth-service` `/me`. If you add a new
   premium feature, register it in
   `services.shared.feature_entitlements` and bump
   `Organization.entitlements_version`.
9. **Preprod validation is the only safe "true reset"**:
   `python3 scripts/preprod_validation.py --mode full-reset` is
   the documented way to get a clean local stack.

## 17. Quick architecture cheat sheet

- **Backend**: Python, FastAPI, SQLAlchemy async, Alembic, Uvicorn.
- **Infra**: MySQL 8 (system of record), Redis 7 (streams/queues/locks,
  `noeviction`), InfluxDB 2.7 (telemetry time-series), EMQX 5.3 (MQTT,
  MySQL auth/ACL), MinIO/S3 (artifacts and datasets).
- **Workers**: separate `APP_ROLE=worker` containers for telemetry
  (`data-telemetry-worker`, `data-telemetry-worker-2`),
  analytics (`analytics-worker`, `analytics-worker-2`),
  reporting (`reporting-worker`),
  waste (`waste-analysis-worker`),
  rules (`rule-engine-worker`),
  device schedulers (`device-service-scheduler`).
- **Backpressure**: explicit per-service thresholds + Redis stream
  `MAXLEN` caps + DLQ + outbox retry/backoff + circuit breaker.
- **Multi-tenant**: every row carries `tenant_id`; SH-tenant hard-cuts
  ran across 4 services; inter-service calls propagate tenant via
  headers + `INTERNAL_SERVICE_SHARED_SECRET`.
- **Frontend**: Next.js 16 + Tailwind + Recharts; Expo Router mobile
  with Skia charts.
- **Observability**: opt-in `monitoring` profile (Prometheus +
  Alertmanager + Grafana + exporters), per-service `/health` and
  `/metrics`, worker healthchecks via Redis ping.
- **CI/CD**: GHCR on `main`; `APP_IMAGE_TAG=sha-<merge-sha>` for
  release; `server-build.yml` is the documented emergency fallback.
- **Validation**: tiered fast → truth-parity → smoke E2E → full
  certification; `preprod_validation.py --mode full-reset` is the
  canonical local reset.
